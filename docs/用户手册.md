# dm-seek（马冬梅计划）用户手册

> **面向研发全角色的「代码现实 × 需求演进」追溯系统**。用户一句自然语言疑问，AI 团队自动完成知识库线索 → 代码定位解读 → Git 时间线 + 工单号抽取 → Jira 业务原因 → 综合结论 → 证据校验，交付带出处的演变报告。
>
> 引擎 = Claude Code，跨 Windows / macOS。

## 1. 产品概述

### 1.1 一句话定位

当你想知道"这段代码为什么这么写"、"这个功能是什么时候改的"、"那个 Bug 的根源是什么"——dm-seek 用一句话回答你的疑问，并给出每个结论的出处证据（代码 / commit / Jira 工单）。

### 1.2 适用人群

| 角色 | 典型场景 |
| --- | --- |
| 开发者 | 调试、溯源、理解遗留代码 |
| 测试 | 确认缺陷根因、验证修复范围 |
| 产品经理 | 了解某个功能的设计决策历程 |
| 架构师 | 跨团队变更影响分析 |
| 新人 | 快速上手模块知识 |

### 1.3 系统概览

dm-seek 是一个由 **7 个 AI agent（专家）组成的团队**，每个 agent 负责一个专业领域，通过共享任务列表 + 消息协作完成从疑问到报告的完整链路：

```
用户提问
    ↓
dongmei-ma（协调者）解析疑问、拆解任务
    ↓
kb-keeper ──→ 查知识库，给线索
    ↓
code-analyst ──→ 定位并解读代码（core-ng）
    ↓
repo-tracer ──→ 提取 Git 时间线 + 抽取工单号
    ↓
jira-tracer ──→ 查 Jira 工单业务原因
    ↓
synthesizer ──→ 综合三源产出结论
    ↓
evidence-verifier ──→ 校验证据 + 给置信度
    ↓
dkongmei-ma 交付带出处、带置信度的报告
    ↓
kb-keeper 将结论沉淀回知识库
```

**校验返工循环**：如果证据不足，自动触发发散返工（最多 2 轮），扩大搜索范围后重新综合。

### 1.4 核心原则

- **以代码为唯一事实基准**：一切结论必须可回挂到具体的代码 / commit / Jira 工单出处
- **所有操作只读**：代码、Git 仓库、Jira 均只读取，不修改
- **置信度分级**：高（三源齐备）/ 中（缺一源）/ 低（推断为主）

## 2. 快速开始

### 2.1 环境要求

- Claude Code（安装并登录）
- Windows 10+ 或 macOS
- （可选）本地 git 仓库 + Git 完整历史
- （可选）Jira 访问权限

### 2.2 下载配置包

将 dm-seek 项目中的 `.claude/` 目录和 `.mcp.json` 文件复制到你的项目根目录。

> **拷贝规则**：运行时只依赖 `.claude/` + `.mcp.json`——只拷这两个即可。`docs/` 是说明文档，不影响运行。

### 2.3 配置凭据（环境变量，零明文）

dm-seek 使用环境变量管理凭据，**绝不将真实 token 写入配置文件**。

**GitHub（多仓支持）**：每个 git 仓库需要一个独立的 GitHub token：

```powershell
# Windows PowerShell
[Environment]::SetEnvironmentVariable("DMSEEK_GH_TOKEN_你的仓库名大写", "你的-token", "User")
```

```bash
# macOS / Linux
echo 'export DMSEEK_GH_TOKEN_你的仓库名大写="你的-token"' >> ~/.zshrc
```

**Jira**：

```powershell
[Environment]::SetEnvironmentVariable("DMSEEK_JIRA_SITE_NAME", "你的-domain", "User")
[Environment]::SetEnvironmentVariable("DMSEEK_JIRA_EMAIL", "你的-邮箱", "User")
[Environment]::SetEnvironmentVariable("DMSEEK_JIRA_API_TOKEN", "你的-token", "User")
```

> 设置完成后**重启 Claude Code / 终端**，环境变量才能生效。你也可以使用引导 skill（`setup-guide`）来自动完成配置——运行 Claude Code 后告诉系统："帮我配置 dm-seek"。

### 2.4 启动团队

在项目目录下运行：

```bash
claude --agent dongmei-ma
```

首次启动时，dongmei-ma 会自动执行一次初始化：
1. 创建团队 `dm-seek`
2. 召唤其余 6 个成员（kb-keeper / code-analyst / repo-tracer / jira-tracer / synthesizer / evidence-verifier）
3. 全部就绪后告诉你"团队已就绪"

> **注意**：自动启动依赖 Claude Code 的 `initialPrompt` 字段。如果不生效（未自动建团），只需在会话内告诉 dongmei-ma："请建团并召唤所有成员"，手动执行初始化的效果完全相同。

### 2.5 提第一个问题

团队就绪后，直接向 dongmei-ma 提一句自然语言疑问：

```
为什么订单超时取消的阈值从 30 分钟改成了 15 分钟？
```

dm-seek 会自动完成全链路分析，几分钟后交付带证据的报告。

## 3. 典型应用场景

### 场景 1：新人快速上手

> "这个订单状态机逻辑在哪？对应的业务规则是什么？"

新人入职，直接通过自然语言询问，系统从 Git 历史 + Jira 自动检索到对应代码模块和业务决策上下文。不需要翻 6 个月的 Slack 和 Confluence，上手时间从天级缩短到分钟级。

### 场景 2：Bug 修复溯源

> "代码有 bug，但 Jira 需求明明是对的……问题出在哪？"

开发者发现 Bug → dm-seek 回源 Jira 取出原始需求 → 对比当前代码实现 → 判断是实现偏差还是需求变更未同步 → 给出修正建议。

### 场景 3：代码审查 + 业务校验

> "这个 PR 的实现符合 Jira 需求吗？"

不只是检查代码风格和语法，还能判断：方法签名改了但关联的 Jira 需求里没有要求改这块——从代码审查升级为业务意图审查。

### 场景 4：跨团队变更影响分析

> "OMS 团队改了 xx 接口，对我们订单模块有什么影响？"

dm-seek 跨仓库追踪，自动分析影响范围，不需要人工梳理调用链路。

### 场景 5：跨地域需求影响评估

> "美国 PM 提了个需求，我们要怎么评估对现有功能的影响？"

典型跨国团队痛点：美国 PM 新需求 → 国内团队评估影响。dm-seek 一键溯源，不再依赖"问资深开发"这种有时差、有负担的方式。

### 场景 6：遗留代码理解

> "这段 3 年前的代码没有注释，当时为什么这么写？"

通过 Git blame → Commit → Jira ticket 链路，自动还原当时的业务决策链。

## 4. 团队角色说明

### 4.1 七名成员一览

| agent | 职责 | 信息源 |
| --- | --- | --- |
| **dongmei-ma** | 你的对话接口。解析疑问、拆解任务、驱动全链路、交付报告 | 不直连信息源（只协调） |
| **kb-keeper** | 知识库管家。查线索、沉淀结论、管理增量知识 | Obsidian 知识库 |
| **code-analyst** | 代码侦探。定位并解读代码，判断 KB 线索的准确性 | 代码文件（本地直读 / 远端经 repo-tracer 取）|
| **repo-tracer** | Git 历史学家。提取时间线、抽取工单号、管理多仓库 | 本地 Git + GitHub MCP（远端独占）|
| **jira-tracer** | 业务分析师。查工单业务原因、因果链 | Jira MCP（只读）|
| **synthesizer** | 分析师。综合代码+Git+Jira 三源产出结论 | 上游三源产物 |
| **evidence-verifier** | 质检员。校验每条结论是否有出处、给置信度 | 上游全部产物 |

### 4.2 dongmei-ma 是你的对话接口

作为用户，你只需要与 dongmei-ma 对话。它负责：

- 接收你的自然语言疑问
- 解析意图（是什么/为什么/影响范围/责任定位等）
- 派发给对应专家成员
- 收集结果 → 交付报告
- 证据不足时触发返工（发散搜索→再综合）
- 默认以**中文**交付报告（按需可出英文）

### 4.3 协作流程

每次查询的典型链路：

1. **kb-keeper** 查知识库 → 给出线索（模块名、文件路径、相关类）
2. **code-analyst** 定位代码 → 解读实现 → 判断 KB 知识的时效性
3. **repo-tracer** 取 Git 时间线 → 从 commit message 抽 Jira 工单号
4. **jira-tracer** 查 Jira 工单详情 → 业务原因 + 因果链
5. **synthesizer** 综合三源 → 产出结论 + executive summary
6. **evidence-verifier** 校验每条结论 → 给置信度
7. 充分则 dongmei-ma 交付 + kb-keeper 沉淀；不足则返工扩大搜索

## 5. 使用方法与输出解读

### 5.1 如何提问

**最佳实践**：越具体越好。

| ✅ 好问题 | ❌ 较模糊的问题 |
| --- | --- |
| "订单超时取消的阈值为什么从 30 分钟改成了 15 分钟？" | "这个系统怎么回事？" |
| "CorporateJobCreationService 的 APPENDABLE_JOB_STATUSES 常量是在哪个 commit 加的？" | "帮我看看这个文件" |
| "DELI-4521 这个工单对应改了哪些代码？" | "有个工单改了点东西" |

### 5.2 输出结构

每份报告有两层：

**第一层：高层结论（executiveSummary）**

面向非技术人员的自然语言摘要。用纯业务语言将 Jira 业务原因与代码变化的高层影响编织为连贯叙事，默认不暴露类名/方法名等代码标识。以一段简述式结尾收官。适合快速阅读。

```
📋 订单超时取消配置发生过一次变更：

2025年8月14日（DELI-4521），订单超时取消阈值从30分钟改为15分钟。
这次变更起源于运营团队提出的缩短库存占用的需求（关联需求 DELI-4400）。

变更落地在订单服务的取消逻辑中，将等待时间从30分钟调整到15分钟。
这是一个单点数值改动，涉及1个代码提交、1个Jira工单，改动范围可控。

综合来看，这次调整本质上是业务策略的优化——运营侧确认15分钟
在用户体验和库存效率之间取得了更好的平衡，由开发侧执行了参数调整。
当前线上订单取消超时策略已以15分钟为准。
```

**第二层：完整推导**

每条结论挂出处（代码行号 / commit SHA / Jira 工单），包含演变时间线、分析路径、未知项。作为第一层的证据支撑。

### 5.3 置信度理解

| 置信度 | 含义 | 说明 |
| --- | --- | --- |
| **高** | 三源齐备且互相印证 | 代码现状 + Git 时间线 + Jira 业务原因 三者闭环，结论可靠 |
| **中** | 缺一源 | 如缺 Jira 业务原因（仅凭 commit message 推断），或仅有 Git 时间线 |
| **低** | 关键结论依赖推断 | 缺直接出处，或三源矛盾无法以代码定论 |

### 5.4 降级交付

当证据链不完整且 2 轮返工后仍不足时，系统不会无限循环或臆造答案，而是：

- 照常出报告
- 明确标注「证据不足」
- 说明具体缺口（缺哪一源、哪一环）
- 给出置信度（此时通常为"低"）

## 6. 配置指南

### 6.1 环境变量清单

| 变量 | 用途 | 必填 |
| --- | --- | --- |
| `DMSEEK_GH_TOKEN_<REPO_SLUG>` | GitHub token（每个仓库一个） | 使用远端模式时必填 |
| `DMSEEK_GH_MCP_URL` | GitHub MCP 端点 URL | 可选，默认使用 Copilot MCP |
| `DMSEEK_JIRA_SITE_NAME` | Jira 站点域名（如 `your-domain`） | 使用 Jira 时必填 |
| `DMSEEK_JIRA_EMAIL` | Jira 登录邮箱 | 使用 Jira 时必填 |
| `DMSEEK_JIRA_API_TOKEN` | Jira API token | 使用 Jira 时必填 |
| `DMSEEK_OBSIDIAN_CLI` | Obsidian CLI 二进制路径 | 使用知识库时必填 |

### 6.2 多仓库配置

dm-seek 支持一次查询横跨多个 git 仓库。配置方式：

1. 每个仓库在 `.mcp.json` 中对应一个独立的 `github-<repoSlug>` 实例
2. 每个实例使用独立的 GitHub token
3. repo-tracer 按 code-analyst 的定位结果自动路由到对应仓库

配置示例（以 `hdr-delivery-project` 为例）：

```json
{
  "mcpServers": {
    "github-hdr-delivery-project": {
      "type": "http",
      "url": "${DMSEEK_GH_MCP_URL:-https://api.githubcopilot.com/mcp/}",
      "headers": {
        "Authorization": "Bearer ${DMSEEK_GH_TOKEN_HDR_DELIVERY_PROJECT}",
        "User-Agent": "dm-seek"
      }
    }
  }
}
```

> 推荐使用引导 skill（`setup-guide`）来自动探测本地仓库、写入配置。在 Claude Code 中运行 `claude --agent dongmei-ma` 启动后，告诉 dongmei-ma 你需要配置，它会引导你完成。

### 6.3 Jira 配置

Jira 使用 `@aashari/mcp-server-atlassian-jira` MCP server，**只读**（仅授予 `jira_get` 工具，不授予写操作工具）。

配置已在 `.mcp.json` 中预留好，只需设置环境变量即可。

### 6.4 本地模式 vs 远端模式

| 模式 | 场景 | 数据来源 |
| --- | --- | --- |
| **态A：无本地仓库** | 非研发用户（PM、测试） | 全程 GitHub MCP 取代码 + 提交历史 |
| **态B：本地仓库未过时** | 研发用户 | 本地直读代码 + 本地 git log（推荐）|
| **态C：本地落后远端** | 代码已更新但本地未拉 | 询问用户：取最新还是用本地 |

> **注意**：过时判定按**文件粒度**，绝不整仓比较。只有与当前查询相关的代码段才检查过时状态。

## 7. 安全与边界

### 7.1 只读政策

**所有对代码、GitHub 仓库、Jira 的操作都是只读的。** 例外只有两项：
1. **`git fetch`**：拉取远端更新本地仓库（用于过时判定）
2. **知识库写操作**：仅 kb-keeper 有权写知识库

### 7.2 独占机制（诚实声明）

dm-seek 对信息源的"独占"是**策略级约束**，不是物理隔离：

- MCP 服务在共享 `.mcp.json` 中配置，**会话层面对全 team 可见**
- 「GitHub MCP 仅 repo-tracer 可用」靠**各 agent 的 tools 白名单**实现
- 独占只针对**远端源类 MCP**；本地 git 无远端凭据风险，读取权共享

**三道防线**：

| 防线 | 机制 | 说明 |
| --- | --- | --- |
| L1 tools 白名单 | 各 agent 的 tools 声明 | 只有授权的 agent 能调用对应 `mcp__` 工具 |
| L2 边界声明 | 每 agent 固定声明区块 | 角色自律约束越界 |
| L3 校验层 | evidence-verifier 产出校验 | 越界可被发现、可审计 |

### 7.3 凭据安全

- 所有凭据使用**环境变量**（`${VAR}`），**绝不将真实 token 写入任何配置文件**
- `.mcp.json`、agent `.md`、settings 中只出现 `${DMSEEK_*}` 占位
- 引导 skill 全程不接收、不存储任何 token 明文
- 请在 `.gitignore` 中添加 `.mcp.json` 以防误提交

### 7.4 Jira 只读

jira-tracer 仅授予 `jira_get` 工具，杜绝任何写/修改工单的操作。这是系统层面的硬限制，不是自律约束。

## 8. 常见问题

### Q1: 启动后没有自动建团

如果 `claude --agent dongmei-ma` 启动后没有自动创建团队 + 召唤成员，只需在会话中告诉 dongmei-ma：

> "请建团并召唤所有成员"

它就会手动执行初始化，效果等同自动启动。

### Q2: 某个 agent 无响应

当 code-analyst 或 repo-tracer 收到任务后显示 idle 但无产出输出时：

1. 尝试缩小任务粒度——例如要求"只看这一个文件的行号范围"，而不是要求完整的 code_location_set
2. 如果一再失败，请检查 tools 白名单是否正确（`Read`、`Bash` 等基本工具是否在列）

### Q3: 如何配置多仓库

见第 6.2 节或运行引导 skill（`setup-guide`）。关键点：每仓一个独立的 GitHub token，repo-tracer 按定位结果自动路由。

### Q4: 看不到 Jira 数据

检查环境变量是否已设置并生效：

```powershell
# PowerShell
echo "JIRA_SITE: $env:DMSEEK_JIRA_SITE_NAME"
echo "EMAIL: $env:DMSEEK_JIRA_EMAIL"
```

如果变量值正确但无效，请重启 Claude Code / 终端。

### Q5: 输出是英文

dm-seek 默认产出中文报告。如需英文版，在提问时补充说明"请用英文回答"即可。

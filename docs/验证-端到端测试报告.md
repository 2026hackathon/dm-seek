## 9. per-agent 独占运行时屏蔽（TC-7.6 屏蔽侧）——【机制已正面佐证，live 演示待部署环境】

**状态更新（2026-06-12，team-lead 用真实 `claude` CLI 2.1.165 当场实测）**：命门由「OPEN / 无正面证据」更新为「**核心机制已获真实 CLI 正面佐证；唯独 live mcp 屏蔽演示待部署环境**」。风险较此前「OPEN / 未知」大幅下降——不再是「白名单是否生效」的未知，而是「确认已知机制」。

### 9.1 当场实测（真实 claude CLI，正面发现）
- **方法**：在 dm-seek 目录以 `claude -p --agent repo-tracer --strict-mcp-config --mcp-config <临时 github 配置>` 启动真实会话，令其列出自身可用工具。
- **结果（关键正面发现）**：repo-tracer 会话的可用工具 = **精确的 `[Bash, Read, SendMessage]`，而非默认全量工具集**。
- **结论**：真实 Claude Code **确实强制执行 agent frontmatter 的 `tools` 白名单，且为「包含式」**（agent 只获得它显式声明的工具）。mcp 工具受同一白名单管辖——repo-tracer 必须**显式声明** `mcp__github-hdr-delivery-project` 才能获得它。
  - 这**推翻了**构建期 harness 探针「白名单可能未生效」的疑虑（那次是中途 spawn 路径未施加白名单，非引擎不支持）；真实 CLI 下白名单确实生效。
  - **逻辑推论**：白名单既为「只给声明的」，则 synthesizer / dongmei-ma 等**不声明任何 mcp 工具**者，运行时即不具备任何 `mcp__` 工具 = L1 屏蔽成立。

### 9.2 未能做成的部分（诚实）
想做「mcp 工具真出现、却被白名单挡住」的**负向 live 演示**，但本会话连不上任何 live MCP server：
1. github（Copilot 托管 MCP）用 `ghp_` 经典 PAT 在 headless 下未能连接（多半需 Copilot 订阅 / OAuth）；
2. 本地 filesystem MCP 被权限策略拦截（未授权自行拉取运行外部 npx 包）。
故「live mcp 工具存在时被屏蔽」这一帧**未直接演示**——但它是 §9.1 已证机制（白名单 = 包含式、mcp 受其管辖）的直接推论，非新的未知。

### 9.3 当前状态与待办
- **状态 = 机制已正面佐证（白名单确为包含式且 mcp 工具受其管辖），live 端到端演示待部署环境。**
- **决定性 live 演示**仍建议由部署方在真实环境跑一次（真实 MCP 正常连接 + 受限 teammate 试调未授权 `mcp__` 工具、观测被拒），一键步骤见 `验证-TC-7.6-独占运行时验证步骤.md`。
- **分支预案（不变）**：被屏蔽 / 拒绝 → L1 硬独占成立、命门 CLOSE；能调用成功 → 降级软隔离（L2 声明 + evidence-verifier 边界校验兜底）+ 升级团队 / 用户，**不假装有硬兜底**。

- **运行时（§8~§9）**：baseline（未受限能调会话级 MCP）已坐实并引用；**per-agent 独占的承重机制已由真实 `claude` CLI 当场正面佐证**（§9.1：`--agent` 下 tools 白名单确为包含式、mcp 工具受其管辖）——命门由 OPEN 升为「机制已佐证、live 演示待部署环境」；唯「live mcp 工具被屏蔽」的端到端负向演示因本会话连不上 live MCP 待部署方按配套步骤坐实。

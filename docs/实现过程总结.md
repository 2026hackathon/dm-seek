## 1. 项目缘起

### 1.1 背景

企业级研发团队在日常工作中面临一个普遍痛点：**"这段代码为什么这么写？"** 要回答这个问题，通常需要串联代码内容、Git 提交历史、Jira 工单三方面的信息，涉及多个人和系统，效率极低。

马冬梅计划的目标是：**用户一句自然语言疑问，AI team 自动完成全链路溯源，交付带证据、带置信度的演变报告。**

### 1.2 技术路线选择

项目的最初计划是构建一个自建的 agent 框架（Python 引擎 + 生命周期管理 + 消息通信），但在需求对齐阶段做了一次关键的方向调整：

> **从「自建框架」转向「Claude Code 成品 team」**——利用 Claude Code 原生的 agent team 能力，角色定义用 `.md` 文件描述、协作靠共享任务列表和消息、MCP 做外部数据源。不重复造引擎轮子。

这一决策节省了约 70% 的工程工作量，让项目聚焦在溯源业务逻辑本身。

## 3. 开发方法论

### 3.1 生成器-批评者循环（Generator-Critic Loop）

每个阶段的设计/实现产物在交付用户前，**必须经过 critic 独立评审**：

```
architect/tools-dev/core-dev 产出
    ↓
critic 独立评审
    ├── B 级及以上问题 → 退回修正（收敛后重审）
    └── C 级问题 → 攒清单批处理，不阻塞
```

critic 的最高优先审查项是**诚实声明红线**——不得夸大"已实现/已验证/已生效"的状态。

### 3.2 以文件为真相源

团队曾多次遇到"agent 报告已完成但文件实际未改"的问题。教训：**不信任 agent 的状态报告，亲自 grep 文件确认改动已落盘。** 这一原则在 critic 审查和 tech-lead 验收时被严格执行。

### 3.3 设计先行，契约定稿再实现

实现前必须先产出设计契约（I/O schema、角色归属表、编排循环），经 critic 审查通过后再进入实现。7 个 agent 的通信格式必须一致，避免"各写各的"。

## 4. 阶段一：需求对齐与范围决策

### 4.1 从 HTML 到 PRD

起点是一份 WeChat 传输的 HTML 文档（马冬梅计划构想），涵盖了 9 类应用场景的设想，但并非最终方案。architect 将其提炼为结构化 PRD（产品需求文档）。

### 4.2 10 个开放问题（O1-O10）

需求对齐过程中识别出 10 个关键决策点，通过 tech-lead 与用户的逐轮确认——解决：

| 问题 | 决策 |
| --- | --- |
| **O1-Knowlery 集成** | 无 API，kb-keeper 通过 obsidian CLI + `/ask` + `/cook` 交互 |
| **O2-过时判定** | 按被检索到的代码段粒度，绝不整仓比较；需要时询问用户 |
| **O3-工单号抽取** | 正则 `^([A-Z]+-\d+)[:\s]`，可配置，容错无号，Revert 穿透抽原号 |
| **O4-置信度** | 三级：高/中/低 |
| **O5-返工上限** | 最多 2 轮发散返工，之后降级交付 |
| **O6-多仓库** | 支持，每仓库独立 GitHub MCP 实例 + 独立 token |
| **O7-KB 初始化** | 可选、不设上限；默认全入口点初始化 |
| **O8-MCP 选型** | Jira：`@aashari/mcp-server-atlassian-jira`；GitHub：Copilot 托管 MCP |
| **O9-输出语言** | 默认中文，英文按需附随 |
| **O10-Figma** | 二期能力，仅 UI 相关 + 有 Figma 链接时触发 |

### 4.3 角色职责确认

确认了 7 个 agent 的职责边界（A4 决策），以及两个必要 skill：KB 初始化 + 引导配置。

> T2 对 hdr-delivery-project 样本仓做了实地核验，发现 5 项与 PRD 偏离的实际编码模式（Controller 非 `execute` 字面名、存储双形态、工单号冒号或空格分隔等），设计中全部据实修正。

### 5.2 关键转折：路径 B——teammate vs subagent

在设计进行到一半时，tech-lead 发现了一个关乎架构根本的**行为分叉**：

- **路径 A（subagent 委派）**：agent 的 `mcpServers`/`skills` frontmatter 字段生效，MCP 物理隔离
- **路径 B（teammate 协调）**：那些 frontmatter 字段**被忽略**，MCP 从共享 `.mcp.json` 加载，会话级可见

**用户裁决选择路径 B（teammate 形态）**，这使得"独占"不再是物理隔离，而是**策略级 tools 白名单约束**。这一决策贯穿了后续所有设计：

- `deniedMcpServers` 不能用作 per-agent 兜底（会话级一刀切，误伤合法 agent）
- per-agent 独占只有 L1（tools 白名单）一个硬机制
- 必须配合 L2（声明区块）+ L3（verifier 边界校验）做软隔离

### 5.3 T7：critic 设计评审——有条件通过

critic 对全部设计产物做了逐字审查，提出 B1-B4 四项收敛项：

| 问题 | 收敛结果 |
| --- | --- |
| **B1** 工单号字段名 | 统一为 `ticketIds`/`noTicket`/`isRevert` |
| **B2** schema 文档升级 | 适配 teammate 形态 + 引用 PRD v0.4 |
| **B3** jira 工具路径 | 内联残块清理，统一指向共享 `.mcp.json` |
| **B4** 字段一致性 | 全契约族对齐 |

T7 最终**有条件通过**，放行条件是 TC-7.6 独占命门必须列为实现期强制验证项。

### 6.2 三道防线设计（独占机制）

实现期最重要的架构创新：

```
防线 1（L1）—— tools 白名单（技术层）
  每个 agent frontmatter 的 tools 声明确立互斥：
  - repo-tracer: 仅它含 mcp__github-*
  - jira-tracer: 仅它含 mcp__jira__jira_get（只读）
  - kb-keeper: 仅它含 obsidian/KB 路径
  - dongmei-ma/其他: 不含任何源类 mcp__

防线 2（L2）—— 声明区块（自律层）
  每个 agent 定义含三段固定区块：
  ## 职责范围 / ## 允许使用的 MCP 服务 / ## 边界约束

防线 3（L3）—— evidence-verifier 校验（审计层）
  校验结论时标记「引用了声明范围外数据来源」的边界违规
```

### 6.3 TC-7.6/TC-7.7 承重假设验证

两项最关键的承重假设验证状态（2026-06-16 H3 端到端测试更新）：

| 测试 | 内容 | 当前状态 |
| --- | --- | --- |
| TC-7.6 | L1 tools 白名单对会话级 mcp__ 工具的屏蔽有效性 | **部分有效**（简单 agent 有效，dm-seek 全部 agent（body > ~50 行）不生效——H3 实测证实；独占依赖 L2 + L3 软边界） |
| TC-7.7 | dongmei-ma initialPrompt 自动启动 | **已验证通过** |

**TC-7.6 验证背景**：2026-06-12 team-lead 用真实 `claude` CLI 2.1.165 以 `--agent repo-tracer` 启动，会话可用工具 = 精确白名单（非全量）→ 证实引擎强制 tools 白名单为包含式、`mcp__` 受其管辖（机制正面佐证）。2026-06-16 qa-engineer H3 端到端测试（Claude Code 2.1.177 + gh-mcp）发现：tools 白名单对 body > ~50 行的复杂 agent 不生效——dm-seek 全部 agent 均属此类。`--strict-mcp-config` 可强制屏蔽 mcp 工具，但同时阻止合法的 repo-tracer。当前独占实际依赖 L2 声明层 + L3 evidence-verifier 校验构成软边界。

### 6.4 文件真相源教训

实现过程中两个 task（#11/#14）遭遇了 agent 产出大文件时卡死（message loop 吞掉输出）的问题。解决方式：**将任务分配给隔离的 background specialist，而非在主线程中通过 teammate 协作完成。** 这一教训被固化在团队记忆中（`agent-teams-file-as-truth-source`）。

### 6.5 critic 实现期审查

实现期进行了两轮 critic 审查：

**第一段（实现 vs 骨架 + 单产物审查）**：全部 9 个产物经文件复审（Bash cat 强制读磁盘绕过 harness 缓存），确认真实现而非骨架。发现 1 处一致性缺口补充、修复孤儿 RULES.md 引用。

**第二段（端到端链路一致性审查）**：4 大维度（链路衔接、抽号字段三方一致、出处证据链、返工 hint 两侧一字不差、三道防线端到端）全部咬合。发现 1 个 B 级问题（信封透传声明缺失）已机械补。

### 6.6 最终交付（2026-06-12）

17/17 任务完成，产物：

- 7 个 agent 定义（`.claude/agents/`）
- 4 个技能（`.claude/skills/`）：coreng-recognition / synthesis-core / kb-init / setup-guide

## 7. 阶段四：优化期

### 7.1 文档结构重组（commit `ca377ae`）

用户反馈 `docs/` 目录过大且 agent 运行时引用 PRD。

**改动**：
- 将 8 份 design 文档 + templates + research 从 `docs/` 迁至 `.claude/rules/`
- 新建 `runtime-spec.md`：从 PRD 抽出 agent 运行时真正依赖的规则（§1-§11）
- 7 agent + 4 skill 共 37 处 `PRD §x.x` 引用全部改为 `runtime-spec §N`
- `docs/` 仅保留说明性文档（PRD + 验证报告）
- 新建 `.claude/settings.json` 显式开启 agent team

### 7.2 协作流程优化（commit `09d2490`）

**优化 A——态B 本地 git 直读**：
- code-analyst 在有本地仓库且非过时时，可直接用 Bash 读取本地 git log，无需转交 repo-tracer
- repo-tracer 信任并收口统一产出 `repo_timeline`（抽工单号不变）
- 独占口径收窄为「远端 GitHub MCP 独占 repo-tracer，本地 git 共享」

**优化 B——KB 匹配审视 + 多 agent 增量沉淀**：
- code-analyst 拿到 KB 线索后先读代码，比对 KB 与实际一致性 → `kbAlignment`
- code-analyst/repo-tracer/jira-tracer 各自产出 `kbIncrement`，不自写 KB
- dongmei-ma 终局归并 → kb-keeper 写入 `modules/`/`entrypoints/` 增量区
- evidence-verifier 红线：KB 偏差不下调 confidence、不触发返工

**优化 C——launcher 内化进 dongmei-ma**：
- 删除 dm-seek launcher 中间层
- `claude --agent dongmei-ma` 直接进入，无转述
- dongmei-ma 兼任启动器（一次性建团+召唤，之后回归协调者）
- TC-7.7 验证步骤

### 7.3 只读政策 + 双层输出 + 分片通信（commit `a2e408c`）

- 确立全局只读政策：代码/Git/GitHub/Jira 全部只读，仅 `git fetch` + KB 写例外
- synthesizer 双层输出：`executiveSummary`（面向非技术人员摘要）+ 完整结论推导
- 分片通信机制：列表字段 >5 条时分片发送，dongmei-ma 归并，下游无感知
- `defaultShell=powershell` + Powershell 权限

### 7.4 渐进式加载（commit `ebd0803`）

- 解决 `design-agent-io-schema.md` 60KB 超过 40KB 加载上限的问题
- 拆分为轻量版（3.9KB，启动自动加载）+ 参考版（运行时按需 Read）
- 更新全部 agent/skill/design-doc 引用路径

### 7.5 用户手册产出

基于现有全部项目文档，生成面向用户的完整手册 `docs/用户手册.md`（8 章，覆盖产品概述到常见问题）。

### 7.6 规则文件清理（2026-06-13）

用户反馈 `.claude/rules/` 下规则文件包含了大量需求迭代演进内容（决策记录、偏离点、开放点裁定、critic 编号、任务引用等），这些是内部开发历史，不应交付给最终用户。

**清理内容**：
- 删除 `_research/` 目录（内部预研笔记，自述"非交付物"）
- 移除所有文件元数据表（版本/日期/负责人/状态/"待审视"/"待 critic"）
- 移除决策记录（路径 A vs B 对比表、"形态变更说明"）
- 移除开放点裁定（6 个文件的 §7~§9 裁定记录）
- 移除 critic C* 编号、T* 任务引用、"已裁定/已定稿"标注
- 移除"偏离点①②③④⑤"标注、"与 PRD 表述差异"注记
- 全部 PRD 引用清零（规则文件完全自包含，引用链仅限 rules 之间 + agent/skill）
- "路径 B"措辞全部替换为"teammate 形态"

**同步清理范围**：7 agent 定义 + 4 skill + 2 模板 + 2 README + 全部 10 个 rules 文件，共 24 个文件。critic 独立审视确认：零 PRD 残留、零断裂引用、角色定义与 rules 一致。

**设计原则确立**：规则文件只包含运行时需要的规则本身，不包含"这些规则怎么来的"历史过程。agent 引用链路自包含（rules ↔ agent/skill），不依赖外部文档。

## 10. 教训与洞见

### 10.1 信任但核实

多个 agent 报告"已完成"但文件实际未改动。**以文件内容为真相源，不信任状态标志**——这是 critic 审查和 tech-lead 验收的核心原则。

### 10.2 诚实声明比完美更重要

在 TC-7.6 命门不能完全坐实的情况下，**诚实标注"机制已佐证、live 待部署"远比声称"已通过"更能赢得信任**。critic 全程把关不放宽口径。

### 10.3 承重假设要显式化

独占机制、启动机制是两个最重要的承重假设。它们需要：
1. 在设计阶段识别并记录
2. 有明确的测试项（TC-7.6/TC-7.7）
3. 有证伪后的降级方案
4. 诚实声明口径一致

### 10.4 prompt 体积是代理可用性的关键

code-analyst / repo-tracer 的 prompt 从 85+ 行精简到 ~30 行后，在 agent team 上下文中的可用性显著提升。**agent prompt 不是文档，越短越好。**

### 10.5 分片能力是 agent team 的必备基础设施

大 payload（几十个 location、上百条 commit）在单次消息中传输会撑爆上下文窗口。分片通信机制不是优化，是**必需**。

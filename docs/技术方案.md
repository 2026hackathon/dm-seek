## 1. 系统架构概述

### 1.1 架构总览

```
┌─────────────────────────────────────────────────────────┐
│                     Claude Code 引擎                      │
│  (agent team runtime: 共享任务列表 + SendMessage 通信)     │
└─────────────────────────────────────────────────────────┘
                            │
    ┌───────────────────────┼───────────────────────┐
    │       协调层           │         执行层          │
    │  (dongmei-ma)         │  6 个专职 teammate     │
    │  用户接口              │  kb-keeper / code-analyst
    │  任务拆解/派发          │  repo-tracer / jira-tracer
    │  校验返工循环           │  synthesizer / evidence-verifier
    │  交付报告              │
    └───────────────────────┴───────────────────────┘
                            │
    ┌───────────────────────┼───────────────────────┐
    │       数据源层         │                        │
    ├── Obsidian KB ──────── kb-keeper（独占读写）      │
    ├── 本地代码仓库 ──────── code-analyst（直读）       │
    ├── GitHub MCP ───────── repo-tracer（远端独占）    │
    ├── 本地 Git ────────── repo-tracer + code-analyst
    └── Jira MCP ────────── jira-tracer（只读）        │
    └──────────────────────────────────────────────┘
```

### 1.2 核心原则

- **代码为唯一事实基准**：一切结论必须可回挂到具体的代码 / commit / Jira 工单出处
- **以代码为锚**：代码现实与文档/记忆/工单冲突时，以代码为准
- **三元闭环**：高置信度 = code（当前实现）+ git（演变时间线）+ jira（业务原因）三者齐备且印证

### 1.3 技术栈

| 组件 | 选型 | 说明 |
| --- | --- | --- |
| 运行引擎 | Claude Code（`claude` CLI） | 提供 agent 运行时、teammate 协作、MCP 框架 |
| 团队形态 | agent team（teammate） | 7 平级 teammate，非 subagent 父子委派 |
| 编程语言 | 无 / 配置驱动 | agent 定义 = Markdown frontmatter；skills = 静态规则文件 |
| 代码仓库 | Git（本地）+ GitHub MCP（远端） | 多仓支持，每仓独立实例+独立 token |
| 知识库 | Obsidian + Knowlery 插件 | 经 obsidian CLI 交互，无独立 API |
| Jira | `@aashari/mcp-server-atlassian-jira` | 通用 HTTP 透传型，无语义化工具，仅 `jira_get` |
| GitHub MCP | Copilot 托管 MCP | `https://api.githubcopilot.com/mcp/` |
| MCP 配置 | 共享 `.mcp.json` | teammate 形态下 MCP 会话级共享 |
| 凭据管理 | 环境变量 `${DMSEEK_*}` | 零明文落盘 |

## 2. 运行形态：teammate

### 2.1 形态概述

dm-seek 采用 **agent team teammate 形态**——7 个 agent 为平级 teammate，经共享任务列表 + SendMessage 消息协作。dongmei-ma 是协调者 teammate（非父节点、非 subagent 委派者）。

teammate 形态的关键特征：
- MCP server 写在共享 `.mcp.json`，会话级对所有 teammate 可见
- Skills 放在项目级 `.claude/skills/` 目录
- Agent frontmatter 的 `mcpServers` / `skills` 字段对 teammate 不生效
- `deniedMcpServers` 是会话级一刀切，不能用作 per-agent 兜底

这对架构的影响：
- MCP 无法实现 per-agent 物理隔离 → 独占靠策略级 tools 白名单约束
- per-agent 独占只有 L1（tools 白名单）一个硬机制
- 必须配合 L2（声明区块）+ L3（verifier 边界校验）做纵深防御

### 2.2 teammate 通信模型

```
dongmei-ma ──SendMessage──→ kb-keeper (分配任务)
kb-keeper ──SendMessage──→ dongmei-ma (返回线索)
dongmei-ma ──SendMessage──→ code-analyst (分配任务)
code-analyst ──SendMessage──→ dongmei-ma (返回定位)
...（逐环推进）
```

所有通信经 **`SendMessage` 工具** + **共享任务列表（TaskCreate/TaskUpdate）** 协作。不存在父子关系——每个 teammate 是独立的参与者。

## 3. 7 Agent 详述

### 3.1 角色一览

| # | Agent | 职责 | 允许的工具/MCP | 不含源 MCP |
| --- | --- | --- | --- | --- |
| 1 | **dongmei-ma** | 协调者、用户接口、拆解派发、校验返工、中文交付 | `Read, TeamCreate, Agent, Task*, SendMessage` | ✅ |
| 2 | **kb-keeper** | 唯一 KB 读写口（Obsidian CLI + Knowlery） | obsidian CLI（Bash）+ Knowlery skills | ✅ |
| 3 | **code-analyst** | core-ng 代码定位+解读、KB 匹配审视、本地 git（态B） | `Read, Grep, Glob, Bash, Skill, SendMessage` | ✅（远端经 repo-tracer） |
| 4 | **repo-tracer** | Git/GitHub 网关，统一收口时间线+抽工单号 | `Bash, Read, SendMessage, mcp__github-*` | ❌（独占远端 GitHub MCP） |
| 5 | **jira-tracer** | Jira 业务原因 → `jira_get` 只读 | `Read, SendMessage, mcp__jira__jira_get` | ❌（仅 Jira MCP） |
| 6 | **synthesizer** | 综合 code+git+jira → 结论（9 场景） | `Read, Skill, SendMessage` | ✅ |
| 7 | **evidence-verifier** | 出处校验 + 置信度 + 边界违规校验 | `Read, SendMessage` | ✅ |

### 3.2 dongmei-ma（协调者兼启动器）

作为对外接口，dongmei-ma 承担双重角色：

**启动角色**（一次性）：
- `claude --agent dongmei-ma` 启动时，自动执行初始化
- TeamCreate → spawn 其余 6 个 teammate → 确认就绪 → 回归协调者

**协调角色**（持续）：
- 接收用户自然语言疑问，解析为 `query_plan`
- 生成 `queryId`，维护 `round` 计数器
- 逐环驱动链路，沿途收集各 agent 产物
- 依据 evidence-verifier 的判定决策：交付 | 发散重派 | 降级
- 终局归并三 agent 的 `kbIncrement` → 委托 kb-keeper 沉淀

### 3.3 kb-keeper（知识库管家）

- 不读源码——只通过 Obsidian CLI + Knowlery `/ask` 检索
- 给线索（模块名、路径、核心类、出处引用）
- 接收 `priorConclusion.exists=true` 时可秒答（跳过全链路）
- 终局接收 dongmei-ma 的 `kb_persist_request`
  - `queries/`：权威结论区（`/cook` 写入）
  - `modules/`/`entrypoints/`：增量区（`append`，知识自增长）
- 降级交付时写轻量 `degraded_note` 隔离区

### 3.4 code-analyst（代码侦探）

三种取码途径：

| 模式 | 场景 | 行为 |
| --- | --- | --- |
| 态B（本地非过时） | 研发用户 | `Read`/`Grep`/`Glob` 直读代码；`Bash git log` 读本地 git 历史 |
| 态A（无本地仓） | 非研发用户 | 经 repo-tracer 发 `code_fetch_request` 取远端代码+历史 |
| KB 未命中 | 知识库无线索 | 按关键词源码兜底（Grep/Glob） |

**KB 匹配审视**：拿到 KB 线索后先读实际代码，比对 KB 描述 vs 代码现实 → `kbAlignment`（consistent/partial/stale/contradicted/kb_miss），供 synthesizer 标偏差。KB 偏差 ≠ 结论缺证据，verifier 不下调置信度。

### 3.5 repo-tracer（Git 历史学家）

- 统一收口产出 `repo_timeline`（提交时间线 + 抽工单号）
- 态B 信任 code-analyst 的 `localGitTimeline`，不重复跑 git log
- 远端经 `mcp__github-<repoSlug>__*` 取码 + 远端历史
- 抽号正则：`^([A-Z]+-\d+)[:\s]`，可配置，Revert 穿透
- 多仓路由：按 `reposInvolved` → 映射表 → 对应本地/远端

### 3.6 jira-tracer（Jira 分析师）

- 仅授予 `mcp__jira__jira_get`（只读）
- Jira MCP 是通用 HTTP 透传型，需自拼 REST v3 path
- 因果脉络：取 issue 详情（含 issuelinks + parent）→ JQL 二次拉相邻工单 → 组装因果链

### 3.7 synthesizer（综合分析师）

详见第 8 章「双层输出」。

方法库：`skills/synthesis-core/SKILL.md`（单一 skill，多 method，对应 9 类场景）

### 3.8 evidence-verifier（质量审核员）

- 校验每条结论是否挂出处（code/commit/jira）
- 输出置信度（高/中/低）
- 不足时触发发散返工
- 标注边界违规（`boundaryViolations`）
- **红线**：KB 偏差不下调 confidence、不进 gaps、不触发返工

## 4. 全链路数据流

### 4.1 标准链路（一次查询，round=0）

```
用户 Q
  │
  ▼
dongmei-ma ────→ query_plan (queryId, intent, keywords, language)
  │
  ▼
kb-keeper ──────→ kb_clue_set (hit, clues[], priorConclusion)
  │
  ▼
code-analyst ───→ code_location_set (locations[], reposInvolved,
  │                 localGitTimeline, kbAlignment, kbIncrement)
  ▼
repo-tracer ────→ repo_timeline (timeline[], ticketIdsAll,
  │                 reposCovered, kbIncrement)
  ▼
jira-tracer ────→ jira_reasons (tickets[], causalChain,
  │                 missingTickets, kbIncrement)
  ▼
synthesizer ────→ synthesis (executiveSummary, conclusions[],
  │                 timelineNarrative, sourcesPresent, unknowns)
  ▼
evidence-verifier → verification (verdict, confidence, gaps,
  │                 divergeHints, boundaryViolations, kbNote)
  │
  ├─ sufficient → dongmei-ma 交付 final_report
  │                → kb-keeper 沉淀 queries/
  │
  └─ insufficient → 发散重派 (round+1, 最多2轮)
                     → code-analyst 段重新循环
```

### 4.2 公共信封格式

所有 agent 间消息共用同一信封，确保多轮次可关联：

```json
{
  "queryId": "q-20260613-001",
  "round": 0,
  "from": "code-analyst",
  "to": "dongmei-ma",
  "payloadType": "code_location_set",
  "chunkInfo": { "chunkId": "loc-1", "chunkIndex": 0, "totalChunks": 1 },
  "payload": { ... }
}
```

- `queryId`：dongmei-ma 生成，其余透传
- `round`：dongmei-ma 维护，每次有效发散 +1
- `chunkInfo`：可选，列表字段 >5 条时建议分片

## 5. Agent 间通信协议

### 5.1 传递形态

采 **「自然语言为主 + 结构化字段可无歧义提取」**——agent 在自然语言回复中必须能无歧义地给出每个字段值，但不强制贴 JSON 代码块。

### 5.2 主要载荷类型

| payloadType | 产出方 | 消费方 | 核心字段 |
| --- | --- | --- | --- |
| `user_query` | user | dongmei-ma | rawQuestion |
| `query_plan` | dongmei-ma | kb-keeper | intent, keywords, expectedOutputs, language |
| `kb_clue_set` | kb-keeper | code-analyst | hit, clues[], priorConclusion |
| `code_location_set` | code-analyst | repo-tracer | locations[], reposInvolved, localGitTimeline, kbAlignment, kbIncrement |
| `code_fetch_request` | code-analyst | repo-tracer | repo, filePath, ref |
| `code_fetch_response` | repo-tracer | code-analyst | staleness, content |
| `repo_timeline` | repo-tracer | jira-tracer | timeline[], ticketIdsAll, reposCovered, kbIncrement |
| `jira_reasons` | jira-tracer | synthesizer | tickets[], causalChain, missingTickets, kbIncrement |
| `synthesis` | synthesizer | evidence-verifier | executiveSummary, conclusions[], sourcesPresent, unknowns |
| `verification` | evidence-verifier | dongmei-ma | verdict, confidence, gaps, divergeHints, boundaryViolations, kbNote |
| `final_report` | dongmei-ma | user | currentState, timeline, rootCause, confidence, degraded, gaps |
| `kb_persist_request` | dongmei-ma | kb-keeper | writeMode, conclusions, increments[] |

各 payload 的完整字段定义见 `.claude/rules/design-agent-io-schema-reference.md`。

## 6. 校验返工循环

### 6.1 状态机

```
round = 0 起算
收到 verification：
  ├ verdict=sufficient (高/中)  → 交付 + 沉淀 → 结束
  └ verdict=insufficient
       ├ round < 2 且选到有新增维度的 hint
       │   → 发散重派，round+1，回 code-analyst
       └ 否则
           → 降级交付：degraded=true，声明"证据不足"+标注缺口
```

### 6.2 防空转

每轮发散**必须有相对上一轮的新增维度**。无新增维度的重派 = 无效返工，不计入 round。

### 6.3 发散动作清单（枚举型，可选）

| 动作 | 范围 |
| --- | --- |
| `widen_kb_search` | 扩大 KB 检索范围 |
| `kb_to_source_fallback` | KB 无果时回源码兜底 |
| `expand_code_scope` | 扩大代码覆盖范围 |
| `add_repos` | 涉及新仓库 |
| `extend_git_history` | 扩展 Git 时间窗口 |
| `relax_ticket_regex` | 放宽工单号抽取正则 |
| `chase_linked_tickets` | 追关联工单 |
| `retry_missing_tickets` | 重试缺失工单 |
| `reframe_synthesis` | 切换分析视角 |

## 7. 独占机制：三道防线

### 7.1 背景

teammate 形态下，MCP 写在共享 `.mcp.json`，**会话层面对全 team 可见**。所有 agent 都有能力调用所有 MCP 工具。「独占」不是物理隔离，而是以下三道防线叠加的策略约束。

### 7.2 防线构成

```
防线           机制                       目标                  粒度
──────────────────────────────────────────────────────────────
L1 技术层      tools 白名单              引擎级阻止调用           per-agent
               各 agent frontmatter 声明   拒绝而非隐藏
L2 声明层      固定边界声明区块             角色自律 + 可审计       每 agent
               ## 职责范围                 违规调用可被识别
               ## 允许的 MCP 服务
               ## 边界约束
L3 校验层       evidence-verifier          检出已发生的越界        运行期
               boundaryViolations[]        标记违规、下调置信度
```

### 7.3 L1：tools 白名单

```
repo-tracer:          tools: [Bash, Read, SendMessage, mcp__github-hdr-delivery-project]
jira-tracer:          tools: [Read, SendMessage, mcp__jira__jira_get]
kb-keeper:            tools: [Read, Grep, Bash, Skill, SendMessage]
code-analyst:         tools: [Read, Grep, Glob, Bash, Skill, SendMessage]
synthesizer:          tools: [Read, Skill, SendMessage]
evidence-verifier:    tools: [Read, SendMessage]
dongmei-ma:           tools: [Read, TeamCreate, Agent, Task*, SendMessage]
```

- **包含式**：白名单列出可用的工具，未列出的工具 agent 无法调用
- 对 `mcp__` 工具同样生效（已由真实 CLI 正面佐证）
- bash 仅限 git 只读操作（态B 本地 git 直读）

### 7.4 关键归属

| 资源 | 独占方 | 非独占方 |
| --- | --- | --- |
| GitHub MCP（远端） | repo-tracer（独占） | 其他 agent 禁调 |
| Jira MCP | jira-tracer（只读独占） | 其他 agent 禁调 |
| 本地 Git 历史 | — | code-analyst + repo-tracer（共享） |
| KB 读写 | kb-keeper（独占） | 其他 agent 仅产物上报（不自写） |

### 7.5 诚实声明

> 本系统的「独占」= L1 白名单 + L2 声明区块 + L3 evidence-verifier 校验三者叠加。L1 白名单对 `mcp__` 工具的屏蔽机制已通过运行验证。

## 8. 双层输出

synthesizer 每次综合必须产出两层，缺一不可。

### 8.1 第一层：高层结论（executiveSummary）

面向**非技术人员（产品经理/测试）**的自然语言结论摘要：

```
📋 订单超时取消配置发生过一次变更：

2025年8月14日（DELI-4521），订单超时取消阈值从30分钟改为15分钟。
这次变更起源于运营团队提出的缩短库存占用的需求（关联需求 DELI-4400）。

变更落地在订单服务的取消逻辑中，将等待时间从30分钟调整到15分钟。
这是一个单点数值改动，涉及1个代码提交、1个Jira工单，改动范围可控。

综合来看，这次调整本质上是业务策略的优化——运营侧确认15分钟
在用户体验和库存效率之间取得了更好的平衡，由开发侧执行了参数调整。
当前线上订单取消超时策略已以15分钟为准。
```

- 纯业务语言描述：**发生了什么 / 为什么 / 影响是什么**，将 Jira 业务原因与代码高层影响编织为连贯叙事
- 引用关键证据但不贴原文
- 默认避免技术标识（不暴露类名/方法名/字段名）；代码与 Jira 有出入时例外
- 以一段简述式结尾收官（非单句结论）
- **不分片**——executiveSummary 必须完整一次送达

### 8.2 第二层：完整推导（conclusions[] + timelineNarrative）

每条结论挂出处：

```json
{
  "statement": "订单超时取消阈值于2025-08-14由30分钟改为15分钟。",
  "dimension": "current_state",
  "evidence": [
    {"type": "code", "ref": "order-service/.../OrderTimeoutPolicy.java#L20"},
    {"type": "commit", "ref": "hdr-delivery@a1b2c3d"}
  ]
}
```

- `sourcesPresent{code,git,jira}`：三源齐备性
- `unknowns[]`：未决/推断点

## 9. 只读政策

### 9.1 总则

**所有对代码、GitHub 仓库、Jira 的操作都是只读的。** 例外仅两项：
1. **`git fetch`**：拉取远端更新本地仓库（过时判定所需）
2. **KB 写操作**：归 kb-keeper（认知库沉淀）

### 9.2 具体约束

| 领域 | 允许的操作 | 禁止的操作 |
| --- | --- | --- |
| 代码文件 | Read / Grep / Glob | 修改/创建/删除 |
| Git 仓库 | log / diff / show / cat-file / **fetch** / ls-remote | push / commit / reset / checkout / tag / rebase / stash / rm |
| Jira | `jira_get`（只读） | 创建/修改/删除工单 |
| GitHub MCP | 取代码 + 取提交历史 | 创建/修改 PR、issue、comment |

## 10. 分片通信机制

当 agent 产出的列表型字段（locations / timeline / conclusions / tickets 等）超过 5 条时，启用分片发送。

### 10.1 分片格式

```json
{
  "chunkInfo": { "chunkId": "loc-set-1", "chunkIndex": 0, "totalChunks": 3 },
  "payload": { "locations": [...], ... }
}
```

### 10.2 归并策略

- 接收方（dongmei-ma）按 `queryId + chunkId` 缓存各片
- 末片（`chunkIndex === totalChunks - 1`）到达后合并列表字段
- 合并后的 payload 转发给下游 agent，**`chunkInfo` 已剥离**，下游无感知
- `round` 变更时清空该 `queryId` 的全部缓存分片

## 11. 双源切换与过时判定

### 11.1 三态切换

| 状态 | 场景 | 代码来源 | Git 历史来源 |
| --- | --- | --- | --- |
| 态A | 无本地仓库 | repo-tracer 远端取码（GitHub MCP） | repo-tracer 远端历史 |
| 态B | 有本地仓库且非过时 | code-analyst 本地直读 | code-analyst 本地 git log → repo-tracer 收口 |
| 态C | 本地落后远端 | 询问用户（取最新/用本地） | 同代码选择 |

### 11.2 过时判定

- **按被检索到的相关代码段粒度**，绝不整仓比较
- 每次只对相关文件按 path 查远端 sha
- `staleness=fresh` → 使用本地；`staleness=stale` → dongmei-ma 询问用户
- dongmei-ma 是**唯一向用户发起询问者**，可合并询问（列出所有 stale 文件）

- teammate 形态下，agent frontmatter 的 `skills` 字段**不生效**
- skills 放在项目级 `.claude/skills/` 目录
- agent 通过 `Skill` 工具调用或 `Read` 直读 SKILL.md

### 12.3 单一权威载体

核心识别规则集中在对应 skill 的单一文件（`SKILL.md`），不散落在各 agent prompt。扩展到其他框架时只新增规则段 + 枚举值，不改契约、不散落到各 agent。

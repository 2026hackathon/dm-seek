一、团队组成：两层结构

1.1 Builder Team（6 人，开发 dm-seek 配置包的团队）

协调 tech-lead

技术协调者。任务拆解与分配、进度跟踪、协调成员间协作、向用户汇报里程碑。是整个 builder team 的调度中枢。

设计 architect

架构师。PRD 撰写与版本管理、架构设计决策、设计文档之间的一致性保证、对设计变更做影响分析和收敛。

实现 core-dev

核心开发者。7 个 agent 定义的编写与修改、运行时规则（runtime-spec）落地。工作对象：.claude/agents/*.md + .claude/rules/。

实现 tools-dev

工具开发者。4 个 skill 实现、.mcp.json 配置模板、配置占位模板维护、引导流程（setup-guide）开发。

质量 qa-engineer

质量工程师。测试计划制定、端到端验证执行、静态核查、问题追踪与回归确认。

把关 critic

评审者。对抗验证（adversarial review）、设计/实现产物审查、边界违规检测、generator/critic 循环的驱动方。≤2 轮辩论，不收敛时升级用户裁决。

1.2 产品 Team（dm-seek 配置包定义的 7 个业务 Agent）

这 7 个 agent 是 builder team 开发出来的产物——一套可导入、开箱即用的 Claude Code agent team 配置包。用户导入后直接使用。

🏗️ dongmei-ma

协调者。解析用户自然语言疑问、拆解子任务并派发、驱动校验返工循环、默认以中文交付报告。不直连任何信息源。

MCP：无（不直连信息源）

📚 kb-keeper

唯一 KB 读写口。经 Obsidian CLI + Knowlery 提供线索、沉淀结论。不读源码，写操作唯一收口。

MCP：无（KB 经 obsidian CLI / Knowlery）

🔍 code-analyst

据 KB 线索定位并解读 core-ng 代码。KB 未命中时源码兜底。态B 可经 Bash 直读本地 git 历史。远端代码经 repo-tracer 获取。

MCP：无（本地直读 + 远端经 repo-tracer）

📜 repo-tracer

Git/GitHub 网关。统一收口提交时间线、从 commit subject 抽取 Jira 工单号。管理 N 个按仓划分的 GitHub MCP 实例。

MCP：mcp__github-*（全部 GitHub 实例，远端独占）

🎫 jira-tracer

经 Jira MCP 取工单业务原因与因果脉络。只读权限（仅 jira_get），自己拼 REST v3 路径。

MCP：mcp__jira__jira_get（只读）

🧠 synthesizer

综合 code + git + jira 三源 → 双层结论（executiveSummary + 完整推导）。分析方法沉淀为可复用 skill，覆盖 9 类场景。

MCP：无（仅消费上游三源产物）

✅ evidence-verifier

校验每条结论是否挂出处 + 三级置信度（高/中/低）+ 边界违规检测。不足时触发 ≤2 轮发散返工，仍不足则降级交付。

MCP：无（仅消费上游全部产物）

3.1 四大配合机制

1

共享任务列表驱动

所有工作通过 TaskCreate / TaskUpdate 透明协作。tech-lead 拆解大任务为子任务，设置 blockedBy 依赖链（设计先行 → 实现 → 审查），成员各自认领执行。你随时可看到进度和阻塞项。

2

分层决策

实现细节：成员自主。设计选择：tech-lead 协调 + critic 审查。架构/方向级：architect 提案 → 你确认 → 执行。关键风险：critic 如实暴露，不包装。

3

Generator / Critic 循环

成员产出 → critic 审查找问题 → 打回修改 → 再审查。最多 2 轮辩论，僵持不下时升级给你裁决。C 级/文档微调攒清单批处理，不打断 owner 关键路径。

4

以文件为真相源

不以任务状态 flag 或口头"已完成"为准，以产物文件的 mtime/size/内容核进度。critic 亲自 grep/Read 核实，不采信二手声称。

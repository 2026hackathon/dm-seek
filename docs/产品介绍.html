1. 产品概述

1.1 解决什么问题

一个功能的完整认知天然分散在四个互相割裂的信息孤岛中。当任何一方与「某人的记忆」产生认知偏差时，就会引发沟通摩擦、决策失误、责任争议。

📁
代码库
它现在「是什么样」
只有开发能读懂
🕰️
Git 历史
「怎么一步步变成今天这样」
散落在 commit log 里
🎫
Jira 工单
每次变更「为什么发生」
淹没在工单海洋中
🎨
Figma（二期）
当初的「设计意图」
与实现脱节

1.2 核心能力

dm-seek 是一个由 7 个 AI Agent 组成的智能团队，以 Claude Code 为运行引擎，在你输入一句自然语言疑问后，全自动完成：

📚 知识库线索 → 🔍 代码定位解读 → 🕰️ Git 时间线 → 🎫 工单号抽取 → 📋 Jira 业务原因 → 🧠 综合结论 → ✅ 证据校验

最终交付一份有出处、带置信度的功能演变报告——每个结论都挂着具体的代码行、commit hash、Jira 工单号。

1.3 核心设计原则

🏛️ 以代码为唯一事实基准

一切结论必须能回挂到具体的代码 / commit / Jira 出处；当「代码现实」与「记忆/文档」冲突时，以代码为锚。

🔒 所有操作只读

代码、Git 仓库、Jira 均只读取不修改。仅 git fetch 和知识库写为例外。

🌱 知识自增长

每次查询结论自动沉淀回知识库，同类问题下次秒答，越用越聪明。

🔄 证据不足则发散返工

不臆造结论。最多 2 轮扩大搜索；仍不足则降级交付并显式声明"证据不足"。

dm-seek 面向研发全角色——不只是开发者，还包括产品、测试、架构等所有需要理解「代码现实」的人。

角色典型痛点dm-seek 提供

🧪 测试工程师发现 Bug 但不确定是「实现偏差」还是「需求变更未同步」代码 vs Jira 原始需求对比

2.1 两种使用模式

🏠 态B — 本地仓库模式

适用：开发者、架构师（有本地 git 仓库）

直读本地代码 + 本地 Git 历史，过时自动询问是否切换远端

☁️ 态A — 远端仓库模式

适用：PM、测试、非研发用户（无本地仓库）

全程经 GitHub MCP 获取代码与提交历史，无需本地环境

场景举例

🌱 场景 A：新人快速上手

"这个订单状态机逻辑在哪？对应的业务规则是什么？"

从 KB 找线索 → 定位代码 → 拉 Git 历史 + Jira → 组装出模块设计决策历程。不需要翻 6 个月的 Slack 和 Confluence，上手时间从天级缩短到分钟级。

🌍 场景 B：跨地域需求影响评估

"美国 PM 提了个需求，我们要怎么评估对现有功能的影响？"

跨国团队典型痛点。一键追溯代码 + Git 变更模式 + Jira 历史，输出影响范围与隐性耦合——不再依赖"问资深开发"。

🏚️ 场景 C：遗留代码理解

"这段 3 年前的代码没有注释，当时为什么这么写？"

通过 Git blame → Commit message → Jira ticket 链路自动还原当时的业务决策链，解决"知其然不知其所以然"的长期痛点。

5. 工作流程

5.1 标准查询流程 — 一次完整溯源

  1. 1

    dongmei-ma 解析疑问

    拆解意图（是什么 / 为什么 / 影响范围 / 责任定位），生成 queryId，派发任务到共享任务列表。

  2. 2

    kb-keeper 查询知识库

    返回线索：模块名、文件路径、核心类、已有结论。若命中已有结论 → 秒答跳过全链路。

  3. 3

    code-analyst 定位并解读代码

    据线索定位具体代码段（core-ng 框架识别）。KB 未命中 → 源码关键词兜底搜索。审视 KB vs 代码现实的匹配度（consistent / partial / stale / contradicted）。态B 直读本地代码 + 本地 Git；态A 经 repo-tracer 远端取码。

  4. 4

    repo-tracer 提取提交时间线

    拉取相关文件的 Git 提交历史。从 commit subject 自动抽取 Jira 工单号（正则 [A-Z]+-\d+，如 DELI-4480）。多仓场景自动路由到对应 GitHub MCP 实例。Revert 提交自动穿透到原始工单。

  5. 5

    jira-tracer 查询业务原因

    逐一获取 Jira 工单详情。追溯关联工单（issuelinks + parent）→ 组装多工单因果脉络。

  6. 6

    synthesizer 综合研判

    依据场景类型选择分析方法（9 场景）。交叉验证三源一致性。以代码为准处理冲突（记录为"记录与实现的偏差"）。双层输出：① executiveSummary（业务语言）② 完整结论（证据链 + 偏差注记）。

  7. 7

    evidence-verifier 证据校验

    每条结论是否挂着代码 / commit / Jira 出处？三源齐备且印证 = 高置信度；缺 Jira = 中；依赖推断 = 低。标记边界违规。证据充分 → 交付 + 沉淀；不足 → 发散返工（最多 2 轮）。

✅ 证据充足
dongmei-ma 交付报告 → kb-keeper 沉淀到 queries/（权威结论区）。知识自增长，下次秒答。
⚠️ 证据不足
发散重派（回到 Step 3 扩大搜索）→ 重新综合。最多 2 轮。仍不足 → 降级交付，明确声明"证据不足"并标注具体缺口。

5.2 知识自增长闭环

📝 终局沉淀

查询完成后，dongmei-ma 委托 kb-keeper 将最终结论写入 queries/（权威结论区）。

🔍 增量沉淀

code-analyst / repo-tracer / jira-tracer 调查过程中发现的增量知识（偏差校正、新入口点、新 commit 线索等），归并后统一写入 modules/ / entrypoints/。

⚡ 下次秒答

同领域问题再次提问时，kb-keeper 直接命中已有结论——跳过全链路，即时返回。

9. 安全与边界

9.1 信息源隔离 — 三道防线

dm-seek 的「源独占」依赖三道防线叠加保证，而非单一物理机制。

L1 · 技术层
🛡️ tools 白名单
各 Agent tools 精确收口。仅 repo-tracer 含 mcp__github-*，仅 jira-tracer 含 mcp__jira__jira_get（只读），其余 Agent 不含源类 mcp__
L2 · 声明层
📜 边界声明区块
每个 Agent 定义含「## 职责范围 / ## 允许使用的 MCP 服务 / ## 边界约束」三段固定区块。硬性禁调领域外 mcp__，跨域经消息向 owner 请求
L3 · 校验层
🔍 运行期审计
evidence-verifier 校验结论时标记边界违规（结论引用声明范围外的工具 / 来源），运行期兜底

9.2 只读政策

所有对代码、GitHub 仓库、Jira 的操作都是只读的。仅有两个例外：

🔄 git fetch

拉取远端更新本地仓库（过时判定所需）

📝 KB 写操作

归 kb-keeper（知识库沉淀）

9.3 诚实声明

🔍 随交付文档发布的诚实声明：
  • 独占是策略级约束，不是进程级物理隔离。MCP 实例在会话层面对全 Team 可见，靠 tools 白名单限制谁能调用
  • per-agent 独占只有 L1（tools 白名单）一个原生机制，已通过真实环境运行验证；即便极端情况下 L1 不生效，声明层 + 校验层仍保证边界清晰、可审计
  • 凭据零明文：任何配置文件只允许 ${DMSEEK_*} 占位

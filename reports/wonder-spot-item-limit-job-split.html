Wonder Spot Order 如何基于 order item limit 把 Task 拆分到多个 Job

queryId q-20260613-001 置信度 高 返工轮次 0 数据源 代码 + Git + Jira（三源闭环互证） 日期 2026-06-13

一句话结论：每个 job 有一个 item 数量上限 wonderSpotMaxOrderItemCount（默认 10，dispatch-portal 可配）。系统先按「同企业 corporateId + 同配送时间窗」把 task 分组，组内按该上限做装箱——能塞进已有未满 job 的就追加（append），塞不下的按 ceil(组内总item / 上限) 新建若干 job；单个 task 过大时还会被拆开跨 job。

① item limit 的数值与配置来源

  • 字段：HDRParentChildTaskSetting.wonderSpotMaxOrderItemCount（Integer） codeHDRParentChildTaskSetting.java#L15
  • 默认值 = 10 codeHDRDeliverySettingBuilder.java#L142
  • 来源 = HDR 配送设置（DB 持久化），dispatch-portal 可改，变更有审计 codeHDRDeliverySettingV2ServiceBuilder.java#L206 codeParentChildTaskSettingHistoryProducer.java#L33-34
  • 固化时机：job 创建那一刻把该值固化进 JobAmount.corporateItemLimit，之后 append 以固化值为准（注释 "determined by creation moment"） codeJobAmount.java#L28-29 codeAppendJobContext.java#L46

⚠ 区分两个上限：通用「Max Child Task per Parent」默认 5（按子任务条数）；Wonder Spot 「item limit」默认 10（按订单 item / 菜品项数量）。用户问的拆单走的是后者（item 维度）。

② 拆分逻辑的代码位置

主控：CorporateJobCreationService.createJob codejob/service/creation/CorporateJobCreationService.java#L88-104

L92 按 CorporateTaskGroupKey(corporateId, groupNumber) 分组 (groupingBy) L94 tryAppend(...) ── 先并入已有 job ├─ appendToPreAllocatedJobs (L325, Mongo PreAllocatedJob) └─ appendToTrulyCreationJobs (L265, MySQL) L96-98 若 append 后无剩余 task → return（不新建） L99-101 有剩余 → buildContext(PRE_CREATE_MULTI_CORPORATE, L165) → allocateAndCreateWithContext(L163) ── 新建 job
  • 分组键 CorporateTaskGroupKey(corporateId, groupNumber)（嵌套静态类） codeCorporateJobCreationService.java#L418-439
  • groupNumber = "{配送时段起}_{止}"（纽约时区 yyyyMMddHHmmss）——同企业 + 同配送时间窗才同组，跨组绝不并入同一 job codeWonderSpotAppendTaskBuilder.java#L62
  • 准入门槛 item_count < corporate_item_limit：truly-created 侧 SQL codeCorporateJobCreationService.java#L278；pre-allocated 侧 codePreAllocatedJobBaseService.java#L51 （叠加 JobStatus / OrderStatus / created_time≥今日EST 过滤）
  • 并发乐观锁 jobRepository.partialUpdate(job, "job_status IN (APPENDABLE_JOB_STATUSES)", ...)，冲突回滚 codeCorporateJobCreationService.java#L295

③ 拆分算法（决定切成几个 job）

阈值推导 CreateJobUtil.calcCorporateMultiTaskMaxItemCountThreshold codeCreateJobUtil.java#L22-29

jobMaxItemCount = wonderSpotMaxOrderItemCount // 默认 10 totalItemQuantity = Σ taskParam.taskAmount.itemCount // 组内总 item partCount = total ≤ jobMax ? 1 : total / jobMax (HALF_UP) partItemThreshold = max( ceil(total / partCount), jobMax ) partitionCount = ceil(总item / jobMaxItemCount) // 切几个 job (UP)

→ 按组批量切分，非逐 item 即时新开。codeCreateJobUtil.java#L38-39

新建侧装箱 CreateJobContext.partitionTaskParamsForCorporate codeCreateJobContext.java#L209-237

for (i in partitionCount): counter = jobMaxItemCount // 每个新 job 的剩余容量 while (counter > 0 && tasks 非空): 整装头部 task (放得下) → 加入，counter -= 该 task itemCount 否则 splitAndReduceTask(按剩余容量拆尾, L228) → 跨 job 续放

append 侧装箱 AppendJobContext.calc codeAppendJobContext.java#L61-102

  • 最小堆挑「现有 itemCount 最小（最空）」的已有 job 先填
  • minimumJob.itemCount ≥ corporateItemLimit 即停止 append（L69-72）
  • 否则 findBestMatchTask + splitAndReduceTask 按 spare 容量拆

多 job 边界条件

  • ① 跨组绝不并入同一 job（分组键隔离）
  • ② 已有 job 达 corporate_item_limit 即不可再 append
  • ③ append 装不下的剩余 task → 走新建路径按阈值切
  • ④ 企业单（CORPORATE）允许把一个 task 拆开跨多个 job（usingSplit=true，L33）

根因解释（业务原因，Jira 史诗印证）

  • 为什么有这个架构：jira史诗 DELI-1892「Parent-Child Task Structure」——一单拆 parent + 多 child task，按每 parent 最大 child 数拆分。
  • 为什么设 item 上限：jira史诗 DELI-2476「Item-Based Wonder Spot Delivery」——Wonder Spot 配送以菜品项（item）为基准，需为单 job 设可配置的 item 数量上限（DELI-2477）。
  • 为什么按上限拆 job：jira史诗 DELI-2847「Wonder Spot Delivery Creation」——厨房放行时为订单创建配送，拆单阈值算法是其实现环节（DELI-3103）。
  • 为什么后来重构 flow：jira史诗 DELI-4329「Parallel Search 竞速派单」（DELI-4359）。

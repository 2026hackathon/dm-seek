⚠️ 效率改进建议

  1. 规则文件按需加载：8 个 design-*.md 规则文件 + runtime-spec 全部注入上下文（~60K），但本次 defect_locate 场景实际只用到了其中约 3 个。建议按 scenario 动态裁剪。
  2. Agent 通信优化：kb-keeper 和 code-analyst 的消息以 idle notification 而非完整内容送达，导致额外的拉回消息和重复代码搜索。改进消息投递机制可减少 ~20K 协调开销。
  3. Git 输出裁剪：git show 的完整 diff 输出（特别是迁移脚本和 Controller 删除）占据了大量 output token。可按需使用 --stat 先概览再定向读取。
  4. 单 agent 模式可行性：对于代码调查类查询（defect_locate），协调者直接搜索代码 + Git 的效率可能优于等待 6-agent 链路的全流程往返。考虑按 intent 类型选择轻量/完整模式。

核心观察

协调者单点完成全链路：在 agent 通信面临挑战的情况下，dongmei-ma 通过直接调用 Read/Grep/PowerShell 工具自行完成了从代码搜索到 Git 取证到综合报告的全流程。这暴露了当前架构的一个实际张力：契约要求 dongmei-ma "不直连任何信息源"，但现实通信限制迫使协调者越过边界。

轻量查询更适合单 agent 模式：对于 defect_locate 类高度聚焦的代码调查，7-agent 全链路编排的协调开销（~20% token）可能超过其带来的分工收益。建议按 intent 类型分层：代码调查类走轻量模式（协调者直调），跨系统综合类走全链路模式。
